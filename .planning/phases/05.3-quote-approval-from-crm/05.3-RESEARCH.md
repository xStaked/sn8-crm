# Phase 05.3: Quote Approval from CRM - Research

**Researched:** 2026-04-01
**Domain:** CRM quote approval UI + conversation-scoped review APIs + reuse of existing owner-review delivery path
**Confidence:** HIGH

## Summary

Phase 05.3 should be planned as an additive extension of the current `owner-review` flow, not as a new approval system. The durable approval state machine already exists in [`apps/backend/src/ai-sales/owner-review.service.ts`](/Users/xstaked/Desktop/projects/sn8-projects/sn8-wpp-crm/apps/backend/src/ai-sales/owner-review.service.ts): it validates the active draft version, records `QuoteReviewEvent`, updates `CommercialBrief.status`, and on approval it already prepares customer delivery and sends the rendered quote through the same outbound Kapso chain used elsewhere. The planning priority is therefore to expose that state cleanly to the CRM and provide a transport-correct write path from the web UI.

The minimal backend surface is four conversation-scoped operations: one summary read for the inbox, one detail read for the selected conversation, and two write actions for approve/request-changes. The detail payload should be assembled from existing `QuoteDraft` and `CommercialBrief` rows. Use `renderedQuote` as the primary preview body, `draftPayload.summary` as the preview summary fallback, and existing brief fields for decision context. Do not invent a second quote rendering pipeline, do not send the quote from the web app, and do not bypass `OwnerReviewService`.

The UI should fit inside the current shell. Keep the existing three-column layout, keep conversation selection in the query string, and add quote-review affordances where the user already works: a pending-review badge or section in the conversation list, and a review card in the detail pane above the manual-reply composer. This satisfies the phase goal without a new route, modal-heavy flow, or dashboard redesign.

**Primary recommendation:** Add conversation-scoped quote review APIs that delegate to `OwnerReviewService`, expose pending-review metadata on the inbox list, and render a review card in `DetailPanel` that approves or requests changes against an explicit draft `version`.

### Pattern 1: Conversation-Scoped Quote Review Read Model

**What:** Keep CRM quote approval under the existing conversation resource. Add an additive summary shape for the inbox and a detail shape for the selected conversation.

**When to use:** Any CRM surface that is already driven by `conversationId` and query-string selection.

**Recommended API surface:**
```typescript
GET /conversations
// Add optional fields:
// pendingQuote: { version, reviewStatus, updatedAt } | null

GET /conversations/:conversationId/quote-review
// Returns latest actionable draft preview for the conversation
```

**Example DTO shape:**
```typescript
type PendingQuoteSummaryDto = {
  version: number;
  reviewStatus: "pending_owner_review" | "ready_for_recheck";
  updatedAt: string;
};

type ConversationQuoteReviewDto = {
  conversationId: string;
  quoteDraftId: string;
  version: number;
  reviewStatus: string;
  renderedQuote: string | null;
  draftSummary: string | null;
  ownerFeedbackSummary: string | null;
  approvedAt: string | null;
  deliveredToCustomerAt: string | null;
  commercialBrief: {
    customerName: string | null;
    summary: string | null;
    projectType: string | null;
    budget: string | null;
    urgency: string | null;
  };
};
```

### Pattern 2: CRM Writes Delegate to OwnerReviewService

**What:** The CRM should call the same durable backend approval logic already used by WhatsApp review. Do not replicate the update/send sequence inside `ConversationsService` or a controller.

**When to use:** Approval and revision requests from the web UI.

**Recommendation:** Add transport-aware methods or extend existing methods with source metadata, because the current feedback string says `via WhatsApp command`.

**Example:**
```typescript
// Source: apps/backend/src/ai-sales/owner-review.service.ts
await this.ownerReviewService.approveDraft({
  action: OwnerReviewAction.APPROVE,
  conversationId,
  version: dto.version,
  reviewerPhone: user.email, // or user.id
});

await this.ownerReviewService.requestChanges({
  action: OwnerReviewAction.REVISE,
  conversationId,
  version: dto.version,
  reviewerPhone: user.email,
  feedback: dto.feedback,
});
```

**Planning note:** Refactor the audit message generation so the persisted feedback says `via CRM` for web actions and `via WhatsApp command` only for inbound owner messages.

### Pattern 3: Preview Data Comes From Existing Draft + Brief Rows

**What:** Build the CRM preview from data that already exists:
- `QuoteDraft.renderedQuote` for the actual customer-facing quote body
- `QuoteDraft.draftPayload.summary` for a short preview summary
- `QuoteDraft.ownerFeedbackSummary` for the latest requested-change context
- `CommercialBrief.summary`, `projectType`, `budget`, `urgency`, `customerName` for decision support

**When to use:** Read endpoint for the detail pane and any pending queue preview.

**Example query:**
```typescript
const draft = await prisma.quoteDraft.findFirst({
  where: {
    conversationId,
    reviewStatus: { in: ["pending_owner_review", "ready_for_recheck", "changes_requested", "approved", "delivered_to_customer"] },
  },
  orderBy: { version: "desc" },
  include: {
    commercialBrief: {
      select: {
        customerName: true,
        summary: true,
        projectType: true,
        budget: true,
        urgency: true,
      },
    },
  },
});
```

### Pattern 4: Review Card Above the Composer, Not a New Screen

**What:** Keep the current shell intact. In `DetailPanel`, render a quote-review card below the header and above the message history whenever the selected conversation has an actionable draft.

**When to use:** Selected conversation has `pending_owner_review` or `ready_for_recheck`.

**UI content:**
- Status badge with text, not color alone
- Draft version label
- Commercial summary block
- Rendered quote preview in a scroll-contained card
- Comments textarea shown for "Solicitar revisión"
- Primary button `Aprobar y enviar`
- Secondary button `Solicitar revisión`

**Why this fits:** It preserves the current inbox/detail mental model and avoids moving the user to another workflow surface.

### Pattern 5: Versioned Write DTOs to Prevent Stale Approval

**What:** Both approve and request-changes actions must include the active draft `version`.

**When to use:** Any mutating action from CRM approval UI.

**Example DTOs:**
```typescript
export class ApproveQuoteDto {
  @IsInt()
  @Min(1)
  version: number;
}

export class RequestQuoteChangesDto {
  @IsInt()
  @Min(1)
  version: number;

  @IsString()
  @IsNotEmpty()
  @MaxLength(2000)
  feedback: string;
}
```

### Anti-Patterns to Avoid

- **Calling `handleOwnerCommand()` from the CRM:** That path is for WhatsApp parsing and owner ack messages, not web actions.
- **Sending the approved quote directly from the controller:** This duplicates `prepareApprovedCustomerDelivery()` and weakens the single guardrail.
- **Approving without a `version`:** A stale tab could approve the wrong draft after a regeneration.
- **Building a second quote renderer for the CRM preview:** `renderedQuote` already exists; reuse it.
- **Creating a separate CRM approval page before proving inline shell fit:** Extra routing and state syncing are unnecessary for this phase.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Approval state machine | New CRM-specific draft lifecycle | `OwnerReviewService.approveDraft()` / `requestChanges()` | Existing logic already enforces valid states, version checks, audit events, and customer delivery |
| Customer quote delivery | Frontend-triggered send or new outbound method | `prepareApprovedCustomerDelivery()` + existing `MessagingService.sendText()` path | Prevents divergence from the already-implemented delivery contract |
| Quote preview generation | New preview template or frontend formatter | Stored `renderedQuote` + `draftPayload.summary` | The preview should reflect exactly what will be sent |
| CRM freshness | Manual `setInterval` or ad hoc refetch code | Existing SWR polling/mutate pattern | Keeps shell behavior consistent with current inbox/detail updates |

**Key insight:** Phase 05.3 is mostly read-model exposure plus UI composition. The core approval and delivery behavior already exists and should remain single-sourced.

## Common Pitfalls

### Pitfall 1: Wrong Audit Source
**What goes wrong:** CRM approvals call `approveDraft()` directly, but the stored feedback still says `via WhatsApp command`.
**Why it happens:** The current service hardcodes the review source language.
**How to avoid:** Add an explicit `reviewSource: "whatsapp" | "crm"` and `reviewedBy` input or thin CRM wrapper methods.
**Warning signs:** `QuoteReviewEvent.feedback` mentions WhatsApp for actions taken in the browser.

### Pitfall 2: Missing Delivered Timestamp
**What goes wrong:** The draft reaches `delivered_to_customer`, but the CRM cannot show when delivery happened.
**Why it happens:** `approveDraft()` updates `reviewStatus` but does not currently fill `deliveredToCustomerAt`.
**How to avoid:** Set `deliveredToCustomerAt` when the outbound customer send succeeds, and expose it in the read DTO.
**Warning signs:** CRM can show "entregada" but not when it happened.

### Pitfall 3: Stale Draft Approval
**What goes wrong:** A user opens version 2, another revision creates version 3, and the stale tab still approves v2.
**Why it happens:** Write endpoints that use only `conversationId` cannot detect outdated UI state.
**How to avoid:** Require `version` in approve/revise bodies and keep `getLatestDraftForCommand()` as the gate.
**Warning signs:** `NotFoundException` or approval mismatches during concurrent review testing.

### Pitfall 4: UI Cramps the Existing Detail Pane
**What goes wrong:** The preview block pushes the timeline and composer into an unusable scroll trap.
**Why it happens:** Rendering the full quote inline without height control or hierarchy.
**How to avoid:** Make the review card a bounded panel with internal scroll for long quote bodies and keep the composer as a sibling footer.
**Warning signs:** Messages disappear below the fold or the composer is no longer visible on laptop screens.

### Pitfall 5: Breaking Existing `/conversations` Consumers
**What goes wrong:** The inbox integration regresses while adding quote metadata.
**Why it happens:** Changing required fields or response shape instead of adding optional metadata.
**How to avoid:** Extend summary DTOs additively and leave existing fields untouched.
**Warning signs:** Existing `useConversations()` render path fails before quote UI is even wired.

## Open Questions

1. **Should pending quotes appear as a filtered inbox section or only as badges on conversations?**
   - What we know: The current shell has only one list column and one detail pane.
   - What's unclear: Whether the planner wants a distinct "Pendientes" subsection in that list.
   - Recommendation: Keep a single list, sort pending-review conversations first, and show a visible badge/section header without introducing a new route.

2. **What actor string should be persisted for CRM approvals?**
   - What we know: `OwnerReviewService` currently persists WhatsApp-flavored audit text using `reviewerPhone`.
   - What's unclear: Whether the desired CRM audit label is `user.email`, `user.id`, or a generic `crm-owner`.
   - Recommendation: Use the authenticated user email if available; it is human-readable in audit history.

3. **Should quote review state be merged into `GET /conversations` or exposed by a dedicated pending endpoint?**
   - What we know: Additive fields on the current summary DTO are the lowest-UI-friction option.
   - What's unclear: Whether preserving the summary DTO as-is is more important than avoiding a second fetch.
   - Recommendation: Start with additive optional summary fields; fall back to a dedicated endpoint only if contract risk becomes material during planning.
