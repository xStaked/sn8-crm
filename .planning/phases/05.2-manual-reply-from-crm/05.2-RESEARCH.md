# Phase 05.2: Manual Reply from CRM - Research

**Researched:** 2026-03-18
**Domain:** NestJS outbound endpoint + SWR optimistic update + Prisma persistence
**Confidence:** HIGH

## Summary

Phase 05.2 is a full-stack feature with three clear layers: a NestJS POST endpoint, a persistence step, and a React compose UI that updates the message list without a full page reload. All three layers have solid foundations already in place — the `MessagingService`/`KapsoAdapter`/`KapsoClient` chain handles outbound sends today, the `Message` Prisma model already stores both directions, and the frontend `useConversationMessages` SWR hook exposes a `mutate` handle usable for optimistic updates.

The backend work is additive: add a `POST /conversations/:conversationId/messages` action to the existing `ConversationsController`, wire it through a new service method in `ConversationsService`, call `MessagingService.sendText`, then persist the sent message. The `MessagingModule` must be imported into `ConversationsModule` for the DI graph to resolve. The Kapso SDK's `sendText` returns `SendMessageResponse` with `messages[0].id` — this is the external Kapso message id to store as `externalMessageId`.

The frontend work is equally targeted: add a compose area to `DetailPanel`, call `POST /conversations/:id/messages` via `apiFetch`, then call `mutate()` on the SWR key for the messages list to re-fetch (or append optimistically). The existing `useConversationMessages` hook needs to expose `mutate` from SWR so the compose component can trigger a refresh after a successful send.

**Primary recommendation:** Keep the endpoint inside `ConversationsController` (not a new controller), import `MessagingModule` into `ConversationsModule`, and use SWR `mutate` to append the sent message locally before the next polling cycle.

### Pattern 1: NestJS POST Action on Existing Controller

**What:** Add a `@Post('conversations/:conversationId/messages')` action to the already-existing `ConversationsController`. Keep it in the same module — do not create a new controller file.

**When to use:** Adding an action that shares the same resource scope (`/conversations`) and auth guard pattern.

**Example:**
```typescript
// Source: existing ConversationsController pattern + NestJS docs
@ApiOperation({ summary: 'Enviar mensaje outbound' })
@ApiBody({ type: SendMessageDto })
@ApiCreatedResponse({ type: SentMessageResponseDto })
@Post('conversations/:conversationId/messages')
@HttpCode(201)
@UseGuards(JwtAuthGuard)
async sendMessage(
  @Param('conversationId') conversationId: string,
  @Body() dto: SendMessageDto,
) {
  return this.conversationsService.sendMessage(conversationId, dto.body);
}
```

### Pattern 2: Service Method — Send + Persist

**What:** `ConversationsService.sendMessage` orchestrates two steps: call `MessagingService.sendText`, then persist to `Message` with `direction: 'outbound'`. It needs both services injected.

**Key detail:** `MessagingService.sendText` currently returns `Promise<void>` — the underlying `KapsoClient.sendText` also returns `void`. To capture the Kapso `messageId`, `KapsoClient.sendText` must be changed to return `Promise<string>` (the `messages[0].id` from `SendMessageResponse`). The chain must propagate this id upward.

**Kapso SDK response shape (HIGH confidence — verified from `index.d.ts`):**
```typescript
interface SendMessageResponse {
  messagingProduct: "whatsapp";
  contacts: MessageContact[];
  messages: MessageInfo[];  // MessageInfo.id is the Kapso message id (wamid)
}
interface MessageInfo {
  id: string;
  messageStatus?: MessageStatus;
}
```

**Example:**
```typescript
// apps/backend/src/channels/kapso/kapso.client.ts — change return type
async sendText(to: string, body: string): Promise<string> {
  const { client, phoneNumberId } = this.assertConfigured();
  const response = await client.messages.sendText({ phoneNumberId, to, body });
  return response.messages[0].id;  // Kapso wamid
}
```

```typescript
// apps/backend/src/messaging/messaging.service.ts — propagate id
```

```typescript
async sendMessage(conversationId: string, body: string) {

  // conversationId IS the contact phone for outbound direction

    conversationId: normalizedId,
    direction: 'outbound' as const,
    body: created.body,
    createdAt: created.createdAt.toISOString(),
  };
}
```

### Pattern 4: SWR Mutate After Send

**What:** The `useConversationMessages` hook uses `useSWR` but currently does not expose `mutate`. To update the message list immediately after send (without waiting for the 5-second poll), expose `mutate` from the hook's return.

**When to use:** After a successful `POST /conversations/:id/messages` response, call `mutate()` to trigger an immediate re-fetch — or use `mutate(current => [...current, newMessage], false)` for optimistic append.

**Example:**
```typescript
// hooks/use-conversation-messages.ts — expose mutate
const { data, error, isLoading, mutate } = useSWR<ConversationMessageDto[]>(...);
// ...
return { messages, error, isLoading, state, mutate };
```

```typescript
// detail-panel.tsx — compose area calls mutate after send
const { messages, state, mutate } = useConversationMessages(selectedId);

async function handleSend() {
  if (!body.trim() || !selectedId) return;
  setSending(true);
  try {
    const sent = await apiFetchJson<ConversationMessageDto>(
      `/conversations/${encodeURIComponent(selectedId)}/messages`,
      { method: 'POST', body: JSON.stringify({ body: body.trim() }) },
    );
    // Optimistic append
    await mutate((current) => [...(current ?? []), sent], false);
    setBody('');
  } catch (err) {
    // surface error to user
  } finally {
    setSending(false);
  }
}
```

### Anti-Patterns to Avoid

- **Creating a separate MessagesController:** Adds a new module/file for no reason. The `POST /conversations/:id/messages` action belongs alongside the GET actions in `ConversationsController`.
- **Returning `void` from KapsoClient.sendText and generating a UUID as externalMessageId:** Loses the real Kapso wamid that uniquely identifies the sent message on Kapso's side. Always capture the real SDK response id.
- **Calling `mutate()` before the POST resolves:** Optimistic updates that don't roll back on failure confuse the user. Either optimistically append and roll back on error, or just re-fetch on success (simpler, sufficient for this phase).
- **Blocking the UI with a spinner for the full send duration:** The send is fast (< 1s typical). Show a brief disabled state on the button, not a full-screen loader.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Outbound WhatsApp send | Custom HTTP call to Kapso API | `KapsoClient.sendText` (already built) | Auth headers, error handling, phone number id injection all handled |
| Request body validation | Manual `if (!body)` checks | `class-validator` + `ValidationPipe` | Consistent with existing DTOs; handles maxLength, type coercion, error response shape |
| Post-send message list refresh | Manual `setTimeout` re-fetch | SWR `mutate` | Purpose-built for cache invalidation; avoids race conditions with ongoing polling |
| Auth on new endpoint | Custom token check | `@UseGuards(JwtAuthGuard)` | Already in place for all conversation endpoints |

**Key insight:** The hardest parts of this feature (WhatsApp send, auth, polling) are fully built. The phase is wiring + a compose UI.

## Common Pitfalls

### Pitfall 1: fromPhone for Outbound Message Row

**What goes wrong:** When persisting an outbound `Message`, `fromPhone` needs to be the CRM's own WhatsApp number (the business phone). The service doesn't currently have access to `KAPSO_PHONE_NUMBER_ID` — it only knows the contact phone (`toPhone = conversationId`).

**Why it happens:** The `ConversationsService` only has `PrismaService` injected today. The phone number id is only known to `KapsoClient`.

**How to avoid:** Two options:
1. Inject `ConfigService` into `ConversationsService` and read `KAPSO_PHONE_NUMBER_ID` directly (simple, consistent with how `KapsoClient` does it).
2. Have `MessagingService.sendText` return both `{ kapsoMessageId, fromPhone }` so the caller can persist correctly.

**Recommended:** Option 1 — inject `ConfigService`, read `KAPSO_PHONE_NUMBER_ID ?? ''`. This keeps the service self-contained and mirrors the `KapsoClient` pattern.

**Warning signs:** A persisted outbound message with `fromPhone = ''` will break `getStableConversationId` for outbound direction because it reads `toPhone` for outbound — which would be correct — but the empty `fromPhone` would still be misleading and could break future queries.

### Pitfall 2: ChannelAdapter.sendText Signature Change Breaks KapsoAdapter

**What goes wrong:** Changing `ChannelAdapter.sendText` to return `Promise<string>` requires updating `KapsoAdapter.sendText` and `MessagingService.sendText` as well. Forgetting the abstract class update causes a TypeScript compile error.

**Why it happens:** TypeScript enforces the abstract contract. If only `KapsoClient` is updated but `ChannelAdapter` and `KapsoAdapter` still declare `Promise<void>`, the compiler rejects the mismatch.

**How to avoid:** Update in order: `KapsoClient` → `ChannelAdapter` (abstract) → `KapsoAdapter` → `MessagingService`. Run `pnpm tsc --noEmit` to confirm before testing.

### Pitfall 3: SWR Key Mismatch Breaks Mutate

**What goes wrong:** `mutate` from `useSWR` only updates the cache for the exact same key used in the original `useSWR` call. If the compose component computes the key differently from what `useConversationMessages` uses (e.g., different encoding), the optimistic update silently does nothing.

**Why it happens:** Different URL encoding or leading slash differences produce distinct SWR keys.

**How to avoid:** Export the `buildMessagesEndpoint` helper from `use-conversation-messages.ts` and reuse it in any component that needs to mutate the same key — or simply expose `mutate` from the hook itself (preferred, as shown in Pattern 4).

### Pitfall 4: Missing MessagingModule Import Causes Runtime DI Error

**What goes wrong:** NestJS throws `Nest can't resolve dependencies of ConversationsService` at startup if `MessagingModule` is not imported into `ConversationsModule`.

**Why it happens:** NestJS DI is module-scoped. `MessagingService` is only available in modules that import `MessagingModule`.

**How to avoid:** Add `MessagingModule` to `ConversationsModule` imports. Verify `MessagingModule` exports `MessagingService` (it already does, per `messaging.module.ts`).

### Pitfall 5: UI — Compose Area Layout Breaks Message Scroll

**What goes wrong:** Adding a fixed-height textarea at the bottom of `DetailPanel` can shrink the scrollable message area if flex layout is not set up correctly, causing older messages to be hidden behind the compose area.

**Why it happens:** The current message list div uses `flex-1 overflow-y-auto`. If the compose area does not live outside that flex child, it competes for space.

**How to avoid:** Structure `DetailPanel` as a flex column: header (fixed) + messages area (`flex-1 overflow-y-auto`) + compose area (fixed height, no flex-grow). The compose area must be a sibling of the messages div, not nested inside it.

```tsx
<div className="flex h-full flex-col bg-background">
  {/* header */}
  <div className="border-b ...">...</div>
  {/* messages */}
  <div className="flex-1 overflow-y-auto px-6 py-6">...</div>
  {/* compose — sibling, NOT child of messages div */}
  <div className="border-t border-border px-6 py-4">
    <ComposeArea ... />
  </div>
</div>
```

```typescript
// Source: apps/backend/src/conversations/conversations.service.ts
private getStableConversationId(message: MessageRow): string {
  const participantPhone =
    this.normalizeDirection(message.direction) === 'outbound'
      ? message.toPhone    // ← for outbound, conversationId = toPhone
      : message.fromPhone;
  return this.normalizeParticipantPhone(participantPhone);
}
```
This means when persisting an outbound message: `toPhone = conversationId (contact phone)`, `fromPhone = own business phone`. The conversation read model will correctly include the message because `toPhone === conversationId`.

## Open Questions

1. **Own phone number for `fromPhone`**
   - What we know: `KAPSO_PHONE_NUMBER_ID` is available via `ConfigService`. The `SendMessageResponse.contacts[0].wa_id` may also carry the sender id after a successful send.
   - What's unclear: Whether `KAPSO_PHONE_NUMBER_ID` is the normalized phone number string or just a Meta-assigned numeric id (which may not match `toPhone` formatting in other messages).
   - Recommendation: Use `KAPSO_PHONE_NUMBER_ID ?? ''` for `fromPhone` in the persisted row — it will always resolve `conversationId` correctly via `toPhone` for outbound direction. Flag in a comment that this field's formatting may differ from inbound `toPhone` values.

2. **Error handling when Kapso send succeeds but DB persist fails**
   - What we know: No transaction wraps send + persist today.
   - What's unclear: Whether a silent persist failure (after a successful Kapso send) is acceptable for v1.
   - Recommendation: Let the Prisma error propagate as a 500 — the user sees an error and can retry. The message was already delivered by Kapso even if the local row wasn't saved; for v1 this is acceptable. Do not add a saga/compensation pattern.
