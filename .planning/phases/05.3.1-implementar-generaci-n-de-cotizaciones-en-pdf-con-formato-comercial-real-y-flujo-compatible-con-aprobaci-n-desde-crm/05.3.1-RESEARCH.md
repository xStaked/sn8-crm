## Summary

Phase 05.3.1 should be planned as a document-rendering layer on top of the existing `QuoteDraft` lifecycle, not as a rewrite of quote generation or approval. The repo already has the durable business object needed for this phase: `QuoteDraft` is versioned, already stores `renderedQuote`, `draftPayload`, `templateVersion`, `approvedAt`, and `deliveredToCustomerAt`, and CRM approval already flows through `OwnerReviewService`. The missing piece is a commercial PDF artifact tied to a specific draft version.

The minimal architecture is server-side PDF generation in the backend using `@react-pdf/renderer`, with a small quote-document service that transforms `QuoteDraft + CommercialBrief` into a branded PDF buffer. This fits the current stack better than HTML-to-PDF through Puppeteer/Playwright because the local environment currently has no Chrome/Chromium runtime, the backend has no browser dependency today, and this phase does not need arbitrary HTML/CSS rendering. React-pdf gives a deterministic Node rendering path plus official support for fonts, page wrapping, streaming, and Express usage.

Generate the PDF lazily per `QuoteDraft.version` when the socio opens or downloads it from CRM, then cache the generated artifact in persistence. Do not generate it at draft creation time, because drafts can be regenerated and `QUOTE_TEMPLATE` is still evolving; eager generation would create stale files and wasted work. Do not wait until approval either, because the stated goal is a commercial-format document compatible with CRM review, which means the partner must be able to inspect the actual PDF before approving.

**Primary recommendation:** Add a backend-only quote PDF service based on `@react-pdf/renderer`, generate a cached PDF lazily for the latest actionable `QuoteDraft.version`, expose it through authenticated conversation-scoped CRM endpoints, and keep customer delivery single-sourced inside `OwnerReviewService`.

### Pattern 1: Version-Locked Lazy PDF Generation

**What:** Generate the PDF the first time CRM asks for the active draft version, then reuse the persisted artifact for subsequent reads.

**When to use:** Review/read path for `pending_owner_review`, `ready_for_recheck`, `approved`, and `delivered_to_customer`.

**Recommendation:**
- Lookup latest review-relevant draft exactly as `ConversationsService` already does.
- If that `QuoteDraft.id` already has a persisted PDF artifact, stream it.
- If not, render it, persist it, then stream it.
- When `OwnerReviewService.processRevisionJob()` creates a new version, that new draft simply has no PDF yet; no backfill job is required.

**Why:** This keeps compute proportional to actual review usage and guarantees the reviewed PDF corresponds to the exact `version` the socio sees in CRM.

### Pattern 2: Separate Document Artifact Persistence

**What:** Store PDF artifact data separately from the quote draft body.

**When to use:** As soon as the system needs a durable commercial document instead of transient bytes.

**Recommended schema shape:**
```prisma
model QuoteDraftDocument {
  id            String   @id @default(cuid())
  quoteDraftId   String   @unique
  kind          String   // "commercial_pdf"
  fileName      String
  mimeType      String   // "application/pdf"
  checksumSha256 String
  sizeBytes     Int
  content       Bytes
  generatedAt   DateTime @default(now())
  quoteDraft    QuoteDraft @relation(fields: [quoteDraftId], references: [id], onDelete: Cascade)
}
```

**Why this is preferable to filesystem storage:**
- The project has no object storage abstraction yet.
- Local disk artifacts would be fragile across deploys.
- Expected volume is small enough for Postgres `bytea` at v1 scale.

### Pattern 3: Conversation-Scoped CRM Surface

**What:** Keep PDF access under the existing conversation-scoped review workflow.

**When to use:** CRM detail panel actions for the selected conversation.

**Recommended API surface:**
```text
GET /conversations/:conversationId/quote-review
// additive metadata:
// pdf: { available, generatedAt, fileName, sizeBytes } | null

GET /conversations/:conversationId/quote-review/pdf
// returns application/pdf for the latest review-relevant draft version
```

**Why:** This preserves the current shell mental model and avoids inventing a second quote resource surface.

### Pattern 4: PDF Mapping Uses Structured Quote Data First, Text Fallback Second

**What:** Build the commercial PDF from the richest durable data available.

**When to use:** Every render.

**Mapping order:**
1. `QuoteDraft.draftPayload.structuredDraft.ownerReviewDraft`
2. `QuoteDraft.draftPayload.summary`
3. `CommercialBrief` fields
4. `QuoteDraft.renderedQuote` as fallback body block

**Why:** `renderedQuote` alone is enough for text preview, but a commercial PDF needs explicit sections, titles, assumptions, and pricing blocks when available.

### Pattern 5: Approval Boundary Stays Intact

**What:** PDF generation is read-side infrastructure, not delivery orchestration.

**When to use:** Always.

**Rules:**
- `ConversationsController` may expose PDF generation/download.
- `QuotePdfService` may render and persist artifacts.
- Only `OwnerReviewService` may continue to trigger outbound customer delivery.
- This phase should not alter `MessagingService.sendText()` semantics unless a later phase explicitly changes delivery to attachment/link.

### Anti-Patterns to Avoid

- **Generating the PDF during quote creation:** Regenerations make the artifact stale immediately.
- **Generating the PDF only after approval:** The socio would approve a text draft, not the real commercial document.
- **Saving PDFs to repo-local disk:** Deploy/runtime portability becomes brittle.
- **Building the PDF directly from the frontend:** Breaks auth, auditing, and future reuse.
- **Changing customer delivery to attach the PDF in this phase:** That reopens outbound channel scope and muddies `APPR-05`.
- **Hand-authoring layout with raw `pdf-lib`:** Too low-level for a polished commercial template given this repo’s current needs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Commercial PDF renderer | Manual PDF drawing primitives for every block | `@react-pdf/renderer` | Gives layout, fonts, wrapping, and server rendering out of the box |
| Artifact cache invalidation | Ad hoc timestamp checks across multiple fields | One persisted row per `QuoteDraft.id` | Draft versioning already gives a natural cache key |
| Browser-based PDF infra | Custom Chrome install/bootstrap logic | Node-side renderer without browser dependency | Reduces environment and CI variance |
| Review workflow rewrite | Separate PDF approval state machine | Existing `QuoteDraft.reviewStatus` + `OwnerReviewService` | Current approval flow is already correct and audited |

**Key insight:** The repo already has the correct business boundary. Phase 05.3.1 is a document artifact problem, not an approval-flow problem.

## Common Pitfalls

### Pitfall 1: Rendering From `renderedQuote` Only
**What goes wrong:** The PDF becomes a plain text dump with weak commercial hierarchy.
**Why it happens:** `renderedQuote` is the easiest field to reuse.
**How to avoid:** Prefer `structuredDraft.ownerReviewDraft.sections` when present; treat `renderedQuote` as fallback content, not the full document model.
**Warning signs:** The PDF cannot consistently show cover, scope, assumptions, and pricing sections.

### Pitfall 2: Stale PDF After Owner Revisions
**What goes wrong:** CRM still downloads the previous version’s PDF after a change request/regeneration.
**Why it happens:** Cache key is `conversationId` instead of `QuoteDraft.id` or `version`.
**How to avoid:** Persist exactly one artifact per `QuoteDraft.id`; never reuse across versions.
**Warning signs:** Downloaded PDF version label does not match the review card version.

### Pitfall 3: Hidden Delivery Scope Creep
**What goes wrong:** The plan quietly starts changing WhatsApp delivery to attachment, media upload, or PDF links.
**Why it happens:** PDF generation is confused with customer delivery.
**How to avoid:** Explicitly scope this phase to CRM generation/review artifact access only.
**Warning signs:** Tasks start modifying `MessagingService`, Kapso media APIs, or outbound send contracts.

### Pitfall 4: Browser Runtime Assumptions
**What goes wrong:** The implementation picks Puppeteer/Playwright and then fails in local/dev/CI because no browser runtime is installed.
**Why it happens:** HTML-to-PDF feels familiar for frontend-heavy teams.
**How to avoid:** Prefer a pure Node renderer for v1 unless exact HTML print fidelity is mandatory.
**Warning signs:** Setup docs start requiring Chrome downloads before a single test can pass.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual PDF primitives (`pdf-lib` only) | React-style document renderers with fonts, wrapping, and server APIs | Mature by 2025; current `@react-pdf/renderer` is `4.3.2` | Faster to produce branded, multi-section commercial documents |
| HTML-to-PDF via bundled browser by default | Still viable, but operationally heavier unless the product already depends on headless browsers | Ongoing | Better fidelity, worse runtime simplicity |
| Eager document generation at creation time | Lazy generation with versioned cache for mutable business drafts | Common current pattern for review artifacts | Avoids stale files and unnecessary work |

**Deprecated/outdated:**
- Treating PDF export as a filesystem-only concern in small monoliths: with no object storage and small volume, DB-backed artifact persistence is usually simpler and more reliable here.

## Open Questions

1. **What commercial template is the source of truth?**
   - What we know: `QUOTE_TEMPLATE` is still `pending-owner-template`.
   - What's unclear: exact branding, legal footer, currency formatting, and mandatory sections.
   - Recommendation: Wave 0 of planning should define a locked PDF template contract before polishing layout details.

2. **Should the customer receive the PDF itself or only text for now?**
   - What we know: current outbound delivery is text via `MessagingService.sendText()`.
   - What's unclear: whether Kapso media/file sending is desired in this milestone.
   - Recommendation: keep this phase CRM-only and defer customer PDF delivery to a separate phase unless the user explicitly expands scope.

3. **Should generated PDFs be inline-viewable in the browser or download-only?**
   - What we know: the CRM already has a quote review card; adding an "Abrir PDF" affordance is straightforward.
   - What's unclear: whether mobile CRM usage needs inline embed support.
   - Recommendation: plan `inline` response headers plus a standard download action; do not build a custom PDF viewer first.
