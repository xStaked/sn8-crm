## Phase Boundary

Next.js app shell que conecta con la Phase 1 API: login/logout con httpOnly cookie JWT, protección de rutas con middleware, y un inbox básico (lista de conversaciones con panel de detalle) que fetch desde la API del backend. <decisions>
## Implementation Decisions

### Estructura del monorepo
- Frontend vive en `apps/web` (paralelo a `apps/backend` ya existente)
- Next.js App Router (no Pages Router) — server components, middleware nativo para rutas protegidas
- Tailwind CSS + shadcn/ui como sistema de componentes

### Visual style
- Dark theme obligatorio — paleta negra/oscura, minimalista
- Referencia directa: **shadcn `dashboard-01`** (`npx shadcn@latest add dashboard-01`)
- Sin colores de acento vistosos — UI discreta, funcional, tipo herramienta interna profesional
- Componentes de shadcn como base — no construir desde cero

### Auth UX
- Sesión expirada: redirigir a `/login` silenciosamente (cuando API devuelve 401, middleware intercepta y redirige)
- Post-login exitoso: siempre a `/dashboard` (destino fijo, sin returnUrl en Phase 1.1)
- Credenciales incorrectas: mensaje de error genérico — "Credenciales incorrectas" sin especificar qué campo falló (seguridad)
- Cookie httpOnly manejada por el backend — el frontend no accede al token directamente, solo hace fetch a los endpoints de auth

### Inbox layout
- Layout de 3 columnas: sidebar de navegación angosta | lista de conversaciones | panel de detalle activo
- Referencia visual: WhatsApp Web / Linear (sidebar + split panel)
- Lista de conversaciones: cada item muestra nombre del contacto + último mensaje (preview truncado) + tiempo relativo
- Panel de detalle: al hacer click en una conversación, muestra en el panel derecho (no navega a otra página)
- Inbox conecta a API real desde el inicio — si el endpoint no existe aún, usar datos mock en el mismo shape esperado

### Claude's Discretion
- Nombres exactos de rutas (e.g. `/dashboard` vs `/inbox`)
- Estructura de carpetas dentro de `apps/web/app/`
- Estrategia de fetching (server components vs client-side con SWR/React Query)
- Skeleton loading states y estados de error en el inbox
- Exact mobile responsiveness — Phase 1.1 es web-first pero no tiene que ser roto en móvil

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- Ninguno — `apps/web` no existe todavía. Phase 1.1 es greenfield para el frontend.

### Established Patterns
- Backend en `apps/backend` usa NestJS con Prisma — el frontend fetch a sus endpoints REST
- httpOnly cookie se establece en la respuesta del backend al hacer login — el frontend solo necesita enviar `credentials: 'include'` en cada fetch

### Integration Points
- `POST /auth/login` → body: `{ email, password }` → respuesta establece cookie httpOnly `access_token`
- `DELETE /auth/logout` → elimina la cookie del servidor
- `GET /auth/me` → verifica sesión activa (usada por middleware Next.js para proteger rutas)
- `GET /contacts` o `GET /conversations` → lista para el inbox (endpoint a definir en Phase 2, mockeable en Phase 1.1)

</code_context>
