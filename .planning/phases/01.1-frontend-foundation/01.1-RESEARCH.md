## Summary

Phase 1.1 is a greenfield Next.js 14 App Router frontend (`apps/web`) that connects to a fully implemented NestJS backend (Phase 1). The backend exposes three auth endpoints: `POST /auth/login`, `DELETE /auth/logout`, and `GET /auth/me` — all using httpOnly cookie JWT sessions. The frontend never touches the JWT token directly; it just uses `credentials: 'include'` on every fetch.

The two most critical implementation decisions are: (1) the Next.js `middleware.ts` pattern for route protection — it must call `GET /auth/me` on every protected path hit and redirect to `/login` on 401, and (2) the CORS configuration on the backend already uses `credentials: true` with `origin: FRONTEND_URL`, which means the frontend must be at `http://localhost:3000` in development (already configured in `main.ts`). The shadcn/ui `dashboard-01` block is the canonical layout reference — install it first, then adapt to the 3-column CRM shape.

For fetching strategy in the inbox, the recommendation is client-side fetch with SWR (lightweight, no extra config, works well with shadcn patterns). Server Components are not recommended for the inbox list because the selected-conversation state is interactive and requires client-side state anyway.

**Primary recommendation:** Scaffold `apps/web` with `create-next-app`, initialize shadcn dark theme, add `dashboard-01` block, implement middleware.ts route protection calling `GET /auth/me`, build login page, then build the 3-column inbox shell.

**Critical placement:** `middleware.ts` must be at `apps/web/middleware.ts` (root of the Next.js project, not inside `src/`). The middleware.ts root location is required by Next.js App Router.

### Pattern 1: Middleware Route Protection

**What:** middleware.ts intercepts every request to `/dashboard/:path*`, calls `GET /auth/me` with forwarded cookies, and redirects to `/login` on 401.

**When to use:** All protected routes. This is the only approved route guard strategy (not client-side redirect, not layout-level fetch).

**Example:**
```typescript
// apps/web/middleware.ts
import { NextRequest, NextResponse } from 'next/server';

export async function middleware(request: NextRequest) {
  const cookie = request.cookies.get('access_token');

  if (!cookie) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  try {
    const res = await fetch(`${process.env.NEXT_PUBLIC_API_URL}/auth/me`, {
      headers: { Cookie: `access_token=${cookie.value}` },
    });
    if (!res.ok) {
      return NextResponse.redirect(new URL('/login', request.url));
    }
  } catch {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/dashboard/:path*'],
};
```

**Critical pitfall:** In Next.js middleware, `fetch` runs in the Edge Runtime. You cannot pass `credentials: 'include'` — you must forward the cookie manually via the `Cookie` header, as shown above.

### Pattern 2: Login Form with httpOnly Cookie Auth

**What:** Client-side form component POSTs to the backend. On success, the backend sets the httpOnly cookie (the frontend never sees the token). On failure, display the generic error message.

**When to use:** Login page only.

**Example:**
```typescript
// src/components/auth/LoginForm.tsx (Client Component)
'use client';
import { useState } from 'react';
import { useRouter } from 'next/navigation';

export function LoginForm() {
  const router = useRouter();
  const [error, setError] = useState<string | null>(null);
  const [loading, setLoading] = useState(false);

  async function handleSubmit(e: React.FormEvent<HTMLFormElement>) {
    e.preventDefault();
    setLoading(true);
    setError(null);
    const data = new FormData(e.currentTarget);
    const res = await fetch(`${process.env.NEXT_PUBLIC_API_URL}/auth/login`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      credentials: 'include',
      body: JSON.stringify({
        email: data.get('email'),
        password: data.get('password'),
      }),
    });
    setLoading(false);
    if (res.ok) {
      router.push('/dashboard');
    } else {
      setError('Credenciales incorrectas. Verifica tu correo y contraseña.');
    }
  }

  return (/* ... */);
}
```

**Key:** `credentials: 'include'` is required for the backend to set the httpOnly cookie cross-origin. The backend's `enableCors({ credentials: true, origin: FRONTEND_URL })` must match exactly.

### Pattern 3: Logout Action

**What:** A button in the sidebar calls `DELETE /auth/logout` with `credentials: 'include'`, then redirects to `/login`.

**When to use:** Sidebar nav logout icon.

**Example:**
```typescript
async function handleLogout() {
  await fetch(`${process.env.NEXT_PUBLIC_API_URL}/auth/logout`, {
    method: 'DELETE',
    credentials: 'include',
  });
  router.push('/login');
}
```

### Pattern 4: SWR for Inbox Data

**What:** Client component uses SWR to fetch conversations. If the API endpoint doesn't exist yet, the `fetcher` returns mock data in the same shape.

**When to use:** Inbox conversation list.

**Example:**
```typescript
// src/hooks/use-conversations.ts
import useSWR from 'swr';

export type Conversation = {
  id: string;
  contactName: string;
  lastMessage: string;
  lastMessageAt: string; // ISO string
  unreadCount: number;
};

const fetcher = (url: string) =>
  fetch(url, { credentials: 'include' }).then((r) => {
    if (!r.ok) throw new Error('fetch failed');
    return r.json();
  });

export function useConversations() {
  const { data, error, isLoading } = useSWR<Conversation[]>(
    `${process.env.NEXT_PUBLIC_API_URL}/conversations`,
    fetcher,
  );
  return { conversations: data ?? [], error, isLoading };
}
```

### Pattern 5: 3-Column Shell Layout

**What:** The authenticated layout is a 3-column CSS grid or flex layout. Sidebar is fixed-width (56px), conversation list is fixed-width (320px), detail panel takes remaining space (flex-1).

**When to use:** All `/dashboard` routes.

**Example:**
```tsx
// src/app/dashboard/layout.tsx
export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="flex h-screen bg-background overflow-hidden">
      <AppSidebar />                          {/* w-14 (56px) */}
      <div className="flex flex-1 overflow-hidden">
        <ConversationList />                  {/* w-80 (320px) shrink-0 */}
        <main className="flex-1 overflow-auto">
          {children}                          {/* detail panel */}
        </main>
      </div>
    </div>
  );
}
```

### Anti-Patterns to Avoid

- **Client-side-only auth guard:** Do not put auth checks only in `useEffect` or layout components. Middleware is the correct layer — it runs before React renders, preventing flash of protected content.
- **Storing the JWT in JavaScript:** The backend handles the cookie. The frontend must never try to read `document.cookie` for the token.
- **Using `redirect()` in Server Components for auth:** For this project, middleware handles auth redirects. Using `redirect()` in `layout.tsx` is fragile and creates race conditions.
- **`credentials: 'include'` omission:** Every single `fetch` to the backend must include `credentials: 'include'`. A missing instance means the cookie is not sent and the request returns 401.
- **Hardcoding API URL:** Use `NEXT_PUBLIC_API_URL` env var. In dev: `http://localhost:3001`. Never hardcode.
- **Adding `dashboard-01` block without dark theme init:** Run `npx shadcn@latest init` first with dark theme configured before adding the block, or the CSS variables will be wrong.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| UI components | Custom button/input/card from scratch | shadcn/ui components | Locked decision; also handles focus states, dark mode tokens, accessibility attributes |
| Icon set | SVG files or custom icon components | lucide-react (already in shadcn) | Consistent stroke width, tree-shakeable, TS types |
| Cookie reading/writing | `document.cookie` parsing | Backend handles all of it | httpOnly cookies are intentionally inaccessible to JS |
| Form validation | Manual regex / custom validation | HTML5 `required`/`type="email"` + shadcn Input | Sufficient for a 2-field login form |
| Loading skeleton | CSS shimmer from scratch | shadcn Skeleton component | Handles dark mode, correct animation |
| Relative timestamps | Date arithmetic + `setInterval` | `date-fns` `formatDistanceToNow` or Intl.RelativeTimeFormat | Edge cases (pluralization, locale) are non-trivial |

**Key insight:** shadcn/ui components are already wired for dark mode via CSS variables. Building from scratch would require re-implementing the entire token system.

## Common Pitfalls

### Pitfall 1: Middleware fetch fails silently in Edge Runtime

**What goes wrong:** The middleware calls `GET /auth/me` but forgets that the Edge Runtime doesn't support Node.js APIs. The fetch itself works, but things like `process.env` behavior or response body parsing can surprise.
**Why it happens:** Developers assume Node.js environment parity.
**How to avoid:** Keep middleware minimal — only check the cookie presence + a lightweight auth/me call. Return early if no cookie (avoids the round-trip for clearly unauthenticated requests). Never import Node.js-only modules.
**Warning signs:** `DynamicServerError` or `Edge Runtime does not support X` errors at startup.

### Pitfall 2: CORS credentials mismatch

**What goes wrong:** Fetch calls succeed but the cookie is never set (or not sent). The browser silently drops the cookie.
**Why it happens:** Either (a) `credentials: 'include'` is missing on the fetch, (b) the backend `origin` doesn't match the frontend origin exactly (trailing slash, http vs https), or (c) the backend is not running when testing.
**How to avoid:** `NEXT_PUBLIC_API_URL=http://localhost:3001` exactly matches the backend's `FRONTEND_URL=http://localhost:3000`. Both must be set in `.env.local`.
**Warning signs:** Login returns 200 but subsequent protected routes redirect to `/login` immediately (middleware gets no cookie).

### Pitfall 3: shadcn `dashboard-01` block conflicts with custom layout

**What goes wrong:** The `dashboard-01` block generates its own sidebar and layout structure. Directly using it without adapting creates a 2-column layout that conflicts with the required 3-column (56px / 320px / flex-1) shape.
**Why it happens:** `dashboard-01` is a reference/starting point, not a drop-in solution.
**How to avoid:** Use `dashboard-01` as structural inspiration and for the CSS variable baseline. Extract only the sidebar nav pattern and dark CSS variable declarations. Rebuild the column layout from scratch to match the spec.
**Warning signs:** Sidebar width is not 56px; detail panel doesn't fill remaining space.

### Pitfall 4: Next.js workspace not added to monorepo root `package.json`

**What goes wrong:** `apps/web` is created but the monorepo root `package.json` `workspaces` array only includes `"apps/backend"`. Running scripts from root doesn't pick up the frontend.
**Why it happens:** `create-next-app` doesn't modify the parent `package.json`.
**How to avoid:** After scaffolding, add `"apps/web"` to the root `package.json` workspaces array. Also ensure `apps/web` has its own `package.json` with a unique `name` like `@sn8-wpp-crm/web`.
**Warning signs:** `npm -w apps/web run dev` fails with workspace not found.

### Pitfall 5: `router.push` after login doesn't reflect the new cookie

**What goes wrong:** After `POST /auth/login` succeeds and cookie is set, `router.push('/dashboard')` triggers middleware, but the middleware fetch to `/auth/me` seems to fail.
**Why it happens:** The cookie is set by the browser correctly, but if the middleware runs before the browser has processed the `Set-Cookie` response, there's a timing issue. More commonly: the `sameSite: 'strict'` cookie policy means the cookie IS sent on same-origin navigation in Next.js — this is fine. The real issue is usually forgetting `credentials: 'include'` on the login fetch.
**How to avoid:** Confirm the login response includes `Set-Cookie` header in DevTools Network tab before debugging middleware.
**Warning signs:** Infinite redirect loop between `/login` and `/dashboard`.

### Pitfall 6: `middleware.ts` placed inside `src/`

**What goes wrong:** When using `--src-dir` with `create-next-app`, developers place `middleware.ts` inside `src/`. Next.js 14 looks for middleware at the project root (`apps/web/middleware.ts`), not inside `src/`.
**Why it happens:** Logical confusion with the `src/` convention.
**How to avoid:** Place `middleware.ts` at `apps/web/middleware.ts` (same level as `next.config.ts`).
**Warning signs:** Route protection doesn't work at all — protected routes load without authentication.
