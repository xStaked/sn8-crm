# Phase 05.1: Integración real de Kapso y flujo inbound end-to-end - Research

**Researched:** 2026-03-18
**Scope:** Close the gap between the existing Kapso webhook pipeline and the current CRM read APIs so a real inbound message can be proven end-to-end.
**Confidence:** HIGH for current backend/frontend seams; MEDIUM for Kapso payload variance because the current code is only tested against simplified fixtures.

---

## Executive Summary

Phase 05.1 is a gap-closing integration phase, not a greenfield webhook phase.

The repo already has all of the major pieces:

- `POST /webhooks/kapso` exists and validates `X-Webhook-Signature`
- the webhook service already enqueues inbound jobs with Redis-backed deduplication
- `KapsoAdapter.normalizeInbound()` already supports a Meta-style webhook shape and a flattened fallback shape
- `MessageProcessor` already persists `externalMessageId` into `Message` and treats Prisma `P2002` as duplicate success
- `GET /conversations` and `GET /conversations/:conversationId/messages` already project persisted messages into the CRM contract

What is still missing is proof that these pieces line up for a real Kapso inbound delivery shape and that the stored message reliably becomes visible in the CRM surface already wired in Phase 5.

The most important planning insight is this:

1. The webhook controller is already secure enough in principle.
2. The real risk is payload-shape mismatch before the queue boundary.
3. The next risk is missing end-to-end evidence from webhook ingress all the way to the read-model endpoints and frontend shell.

That means the phase should focus on hardening the ingress contract, proving idempotent persistence, and documenting a reproducible local smoke path.

## Key Planning Risks

### 1. Real payload accepted by controller but ignored by webhook service

The controller can accept a correctly signed request while `WebhooksService` still returns `{ status: "ignored" }` if it cannot resolve the message id from the nested payload. This is the first real blocker because it prevents the queue handoff entirely.

### 2. Testing only simplified fixtures

Current tests exercise simplified payloads such as `{ message: { id } }`. That is not enough to claim "Kapso real inbound" readiness. The phase should add fixtures that look like actual Kapso/Meta deliveries.

### 3. Duplicate handling proven only at isolated layers

Redis-backed webhook deduplication and Prisma-backed persistence deduplication both exist, but the phase still needs one narrative proving repeated inbound delivery does not create duplicate `Message` rows in the end state the CRM reads.

### 4. Frontend evidence can be masked by development fallback or stale caches

If the inbox hook quietly falls back to dev data or the dashboard does not refresh after a webhook test, the user can think the integration is broken or, worse, think it is working when it is not. Verification needs to explicitly prove the UI is reading real persisted backend data.

### 5. Overlap with Phase 5 Plan 05-04

`05-04-PLAN.md` already targets end-to-end verification for login, inbox, detail, and logout. Phase 05.1 should not redo generic auth smoke from scratch. It should extend the verification surface with a concrete inbound-message proof:

- signed inbound webhook
- persisted message
- updated read endpoints
- updated CRM views

## Recommended Plan Breakdown

### Plan 1: Ingress hardening for real Kapso payloads

First unblock the queue boundary:

- accept real Kapso payload variants for id extraction
- keep strict signature rejection
- add tests around real inbound fixtures

This is the earliest blocking step because nothing downstream matters if a real payload is ignored before enqueue.

### Plan 2: Idempotent persistence and read-model proof

Then prove the stored inbound message reaches the existing conversation projections:

- no duplicate `Message` rows on repeated delivery
- stable phone-based `conversationId`
- updated `/conversations`
- updated `/conversations/:conversationId/messages`

This plan should stay on the current schema and read-model strategy.

### Plan 3: Local setup and end-to-end CRM verification

Last, package the reproducible runbook:

- `.env`/secret expectations
- signed local simulation path
- optional real Kapso/tunnel path
- dashboard verification steps
- minimal frontend refresh glue only if the current UX hides fresh inbound data

This is where the phase earns its acceptance criteria. Without this evidence, the backend work remains partially inferred.

## Recommended Test Strategy

- Unit/spec test: extract idempotency key from both nested and flattened Kapso payloads
- Unit/spec test: `KapsoAdapter.normalizeInbound()` maps real fixtures into the expected normalized message shape
- Integration/e2e test: repeated inbound delivery yields one durable `Message` row
- Integration/e2e test: persisted inbound row appears in `/conversations` and `/conversations/:conversationId/messages`
- Manual smoke: signed inbound delivery against local backend, then verify inbox/detail in the CRM

If a frontend refresh tweak is needed, validate it against the current shell only. Do not redesign components or change the response contracts.

---

## Local Verification Shape

The safest verification path should support both:

1. **Controlled local simulation**
   - sign a JSON fixture with `KAPSO_WEBHOOK_SECRET`
   - `POST /webhooks/kapso`
   - verify database/API/frontend result

2. **Optional real delivery via Kapso**
   - expose local backend through a tunnel
   - configure Kapso webhook URL + shared secret
   - send a real WhatsApp inbound message
   - verify the same outcome

The controlled simulation path is mandatory because it works today even without live Kapso credentials.
