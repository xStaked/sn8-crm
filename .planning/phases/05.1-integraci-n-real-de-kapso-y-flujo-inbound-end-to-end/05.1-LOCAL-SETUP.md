# Phase 05.1 Local Setup: Kapso Inbound Smoke Flow

This runbook proves the current local path:

`POST /webhooks/kapso` -> inbound persistence -> `GET /conversations` -> `GET /conversations/:conversationId/messages` -> current CRM inbox/detail

Use Flow A when you want a deterministic local proof without live Kapso credentials. Use Flow B when you want the same proof with a real Kapso delivery through a local tunnel.

## Required Environment Variables

Copy [.env.example](/Users/xstaked/Desktop/projects/sn8-projects/sn8-wpp-crm/.env.example) into the env files you use locally and set at least:

- `DATABASE_URL`
- `REDIS_HOST`
- `REDIS_PORT`
- `JWT_SECRET`
- `SEED_USER_EMAIL`
- `SEED_USER_PASSWORD`
- `KAPSO_WEBHOOK_SECRET`
- `FRONTEND_URL`
- `NEXT_PUBLIC_API_URL`

Only the optional real-Kapso flow also needs:

- `KAPSO_API_KEY`
- `KAPSO_PHONE_NUMBER_ID`

## Startup Order

From repo root:

1. Start infra required by the backend.
```bash
docker compose up -d postgres redis
```

2. Start the backend.
```bash
npm --workspace @sn8-wpp-crm/backend run start:dev
```

3. Seed the local CRM user if you do not already have one.
```bash
npm --workspace @sn8-wpp-crm/backend run seed:user
```

4. Start the frontend in a second terminal.
```bash
npm --workspace @sn8-wpp-crm/web run dev
```

The backend listens on `http://localhost:3001` and the frontend on `http://localhost:3000` unless you override the defaults.

## Login Session For API Checks

Create a cookie jar you can reuse for `GET /conversations` and `GET /conversations/:conversationId/messages`.

```bash
curl -i \
  -c /tmp/sn8-kapso.cookies \
  -H 'Content-Type: application/json' \
  -d "{\"email\":\"${SEED_USER_EMAIL}\",\"password\":\"${SEED_USER_PASSWORD}\"}" \
  http://localhost:3001/auth/login
```

The response should be `200` and set an auth cookie. Keep `/tmp/sn8-kapso.cookies` for the rest of the smoke test.

## Flow A: Controlled Local Simulation

### 1. Create a deterministic inbound payload

Use a unique external id each time you run the smoke test.

```bash
cat <<'JSON' >/tmp/kapso-inbound.json
{
  "object": "whatsapp_business_account",
  "entry": [
    {
      "id": "waba-local-smoke",
      "changes": [
        {
          "field": "messages",
          "value": {
            "messages": [
              {
                "id": "wamid.local-smoke-001",
                "from": "573001234567",
                "timestamp": "1735689600",
                "type": "text",
                "text": {
                  "body": "Hola desde el smoke local de Kapso"
                }
              }
            ],
            "contacts": [
              {
                "wa_id": "573001234567",
                "profile": {
                  "name": "Smoke Local"
                }
              }
            ],
            "metadata": {
              "phone_number_id": "local-phone-number-id"
            }
          }
        }
      ]
    }
  ]
}
JSON
```

If you repeat the flow later, change `"wamid.local-smoke-001"` to a new value.

### 2. Sign the raw JSON body with `KAPSO_WEBHOOK_SECRET`

The webhook accepts either hex or base64 HMAC-SHA256. This example sends hex.

```bash
SIGNATURE=$(
  node -e "const fs=require('fs'); const crypto=require('crypto'); const body=fs.readFileSync('/tmp/kapso-inbound.json'); process.stdout.write(crypto.createHmac('sha256', process.env.KAPSO_WEBHOOK_SECRET).update(body).digest('hex'))"
)
```

### 3. POST the signed payload to the current webhook

```bash
curl -i \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Signature: ${SIGNATURE}" \
  --data-binary @/tmp/kapso-inbound.json \
  http://localhost:3001/webhooks/kapso
```

Expected response on first delivery:

```json
{"message":"ok","status":"enqueued","messageId":"wamid.local-smoke-001"}
```

### 4. Re-send the same event to prove idempotency

Run the exact same `curl` again.

Expected response on the second delivery:

```json
{"message":"ok","status":"duplicate","messageId":"wamid.local-smoke-001"}
```

### 5. Verify the updated conversation list

```bash
curl -s \
  -b /tmp/sn8-kapso.cookies \
  http://localhost:3001/conversations
```

Expected result:

- one conversation with `"id":"573001234567"`
- `"lastMessage":"Hola desde el smoke local de Kapso"`

### 6. Verify the updated conversation history

Use the conversation id returned by `/conversations`. For the sample payload above it should be `573001234567`.

```bash
curl -s \
  -b /tmp/sn8-kapso.cookies \
  http://localhost:3001/conversations/573001234567/messages
```

Expected result:

- at least one message with `"conversationId":"573001234567"`
- the new inbound message body appears in chronological history
- repeating the webhook did not create a duplicate copy of the same inbound message

### 7. Verify the same data inside the CRM

1. Open `http://localhost:3000/login`
2. Sign in with `SEED_USER_EMAIL` / `SEED_USER_PASSWORD`
3. In the inbox, confirm the conversation `573001234567` appears with the latest preview `Hola desde el smoke local de Kapso`
4. Select that conversation
5. In the detail panel, confirm the same inbound message body appears in message history

If the message is already visible without a hard reload, no frontend change is required for the smoke flow.

## Flow B: Optional Real Kapso Delivery Via Local Tunnel

Use this when you want a real device message to hit the same local backend.

### 1. Expose the backend publicly

Example with Cloudflare Tunnel:

```bash
cloudflared tunnel --url http://localhost:3001
```

Or use another tunnel provider. Keep the public HTTPS URL.

### 2. Configure Kapso to hit the local webhook

Set the webhook destination in Kapso to:

```text
https://<your-public-url>/webhooks/kapso
```

Use the same `KAPSO_WEBHOOK_SECRET` locally and in Kapso so `X-Webhook-Signature` verification still passes.

### 3. Send a real WhatsApp inbound message

From a real WhatsApp client, send a message to the Kapso-connected number tied to `KAPSO_PHONE_NUMBER_ID`.

### 4. Verify the exact same outcome

Repeat the same checks from Flow A:

- `GET /conversations`
- `GET /conversations/:conversationId/messages`
- CRM inbox preview
- CRM detail history

The real payload path is successful only if all four views reflect the same persisted inbound message.

## Troubleshooting

- `401 Unauthorized` from `/webhooks/kapso`: the signature is missing, malformed, or signed with a different `KAPSO_WEBHOOK_SECRET`
- `{"status":"ignored"}`: the payload does not contain a resolvable inbound message id
- `401` from `/conversations`: re-run the `/auth/login` step and refresh the cookie jar
- No CRM update after the APIs already show the message: verify the frontend is pointing at the backend via `NEXT_PUBLIC_API_URL=http://localhost:3001`

## Evidence Checklist

The smoke flow is complete only when all of these are true in the same run:

- valid signed webhook request returns `status: "enqueued"`
- repeated delivery returns `status: "duplicate"`
- `/conversations` shows the updated conversation summary
- `/conversations/:conversationId/messages` shows the new inbound message once
- the current CRM inbox and detail panel show the same persisted message
