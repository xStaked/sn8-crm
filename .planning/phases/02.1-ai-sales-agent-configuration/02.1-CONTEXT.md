# Phase 2.1: AI Sales Agent Configuration - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning
**Source:** User planning brief + roadmap restructuring

<domain>
## Phase Boundary

Esta fase inserta un agente comercial de IA entre el motor conversacional base y el approval loop formal de cotizaciones.

El entregable no es solo "usar IA", sino este comportamiento completo:

1. El bot base completa la calificación inicial sin IA.
2. La IA asume la conversación como asesor comercial experto en desarrollo de software.
3. La IA profundiza el discovery comercial para cerrar un brief suficientemente bueno para cotizar.
4. La IA arma una cotización preliminar usando el formato definido por el socio.
5. La cotización se consulta internamente al socio.
6. La IA incorpora correcciones hasta obtener aprobación explícita.
7. Solo después de esa aprobación puede continuar el loop operativo que culmina con entrega al cliente.

La fase debe enfocarse en la configuración del agente, sus guardrails, el flujo de consulta interna y la estructura de datos necesaria para que la Fase 3 pueda operar sobre una cotización preliminar ya preparada.
</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- La fase será una inserción `2.1` antes de la Fase 3 actual.
- El agente debe vender `desarrollo de software a medida` como oferta principal de v1.
- El tono del agente debe ser `premium` y `consultivo`; no debe sonar agresivo ni improvisado.
- El agente debe cubrir la mayor parte posible del proceso comercial sin saltarse la aprobación humana.
- El agente debe generar la cotización en un formato definido por el socio; ese formato es un input obligatorio de implementación.
- La cotización siempre se consulta primero al socio y puede iterarse varias veces antes de aprobación.
- Ninguna cotización puede enviarse al cliente sin aprobación explícita del socio.
- El agente debe posicionarse como experto en ventas de desarrollo de software, no como bot genérico de soporte.

### Claude's Discretion
- Definir si la consulta al socio ocurre primero por WhatsApp, por CRM, o por ambos, siempre que quede una ruta principal clara en v1.
- Elegir si la cotización preliminar se representa como JSON estructurado, markdown renderizable o ambos, siempre que el formato final respete la plantilla del socio.
- Definir cuánta memoria conversacional adicional necesita la IA por encima del FSM base, siempre que las transiciones sigan auditables.
- Diseñar los estados intermedios del agente comercial siempre que separen con claridad discovery, drafting, review y waiting-for-approval.
</decisions>

<specifics>
## Specific Ideas

- El agente debe preguntar y consolidar al menos:
  - tipo de proyecto
  - problema de negocio
  - alcance deseado
  - presupuesto
  - urgencia
  - restricciones o condiciones relevantes
- El cliente debe percibir acompañamiento comercial serio, no solo un formulario disfrazado de chat.
- La IA puede preparar una cotización preliminar, pero no debe presentar un precio final como cerrado sin revisión humana.
- La experiencia comercial debe ser coherente con SN8 Labs como agencia de desarrollo a medida.
- La fase probablemente requerirá:
  - prompt system versionado
  - esquema estructurado del brief comercial
  - esquema estructurado de cotización preliminar
  - reglas de guardrail para promesas comerciales
  - mecanismo de feedback del socio para regeneración

Inputs todavía pendientes del socio para la implementación fina:
- plantilla exacta de cotización
- reglas de pricing o sanity bounds específicos
- ejemplos de frases que sí / no representan la voz comercial
- ejemplos reales o simulados de conversaciones
</specifics>

<deferred>
## Deferred Ideas

- Negociación avanzada de descuentos
- Catálogo configurable de servicios y pricing desde el CRM
- Entrenamiento fino o dataset supervisado con histórico real
- Multi-canal más allá de WhatsApp
- Envío final automático de la cotización al cliente
</deferred>
