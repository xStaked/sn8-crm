# Phase 2.1: AI Sales Agent Configuration - Research

**Researched:** 2026-03-19
**Domain:** AI sales-agent orchestration, persistent commercial brief state, draft quotation workflow, and human review guardrails
**Confidence:** HIGH

## Summary

Phase 2.1 is not a thin prompt layer. The current codebase already provides the transport substrate needed to trigger the future agent workflow: Kapso inbound webhooks enqueue into BullMQ, inbound/outbound messages persist in Prisma, and authenticated CRM endpoints can already read and send conversation history. What is missing is the actual sales-agent domain: there is no persistent conversation state beyond raw messages, no structured commercial brief, no quotation draft entity, no AI provider module, and no review loop state that can hold a quotation until the socio approves it.

This means the correct planning shape is:

1. Create persistent domain primitives for AI sales work.
2. Add an orchestration layer that turns Phase 2's completed qualification into structured AI discovery + draft quotation generation.
3. Add an internal review/regeneration loop that explicitly blocks customer delivery until approval.

The project already has several strong reuse points:

- `WebhooksService` and `MessageProcessor` handle inbound transport and durable message persistence.
- `Message` rows already give a chronological source of truth for the conversation transcript.
- `ConversationsService` already computes stable `conversationId` values from phone identity.
- `MessagingService` already sends outbound WhatsApp text and persists manual replies.

The main architectural gap is not channel IO; it is the lack of domain state and orchestration around "what stage is this lead in, what do we know, what quote draft is pending, and what feedback from the socio is still unresolved?" Phase 2.1 should solve exactly that gap and leave Phase 3 to formalize validation/persistence/approval delivery on top of the quote draft already produced here.

## Current Codebase Findings

### What already exists

#### 1. Transport + persistence substrate is in place

- `apps/backend/src/webhooks/webhooks.service.ts`
  - Accepts Kapso inbound payloads.
  - Reserves idempotency keys in Redis.
  - Enqueues jobs into BullMQ queue `incoming-messages`.
- `apps/backend/src/messaging/processors/message.processor.ts`
  - Normalizes inbound payloads through `ChannelAdapter.normalizeInbound(...)`.
  - Persists the normalized message into Prisma `Message`.

This means the AI phase does not need a new ingestion path.

#### 2. Stable conversation identity already exists

- `apps/backend/src/conversations/conversations.service.ts`
  - Derives stable `conversationId` from participant phone identity.
  - Returns chronological conversation history via `listConversationMessages`.

This is the right anchor key for AI brief state and quotation draft state.

#### 3. Outbound WhatsApp delivery already exists

- `apps/backend/src/messaging/messaging.service.ts`
  - Wraps the channel adapter for outbound sends.
- `apps/backend/src/conversations/conversations.service.ts`
  - Already has `sendMessage(...)` for CRM-driven outbound text.

This means the review loop can reuse existing outbound primitives when it needs to notify the socio or send controlled customer messages later.

### What does NOT exist yet

#### 1. No AI provider integration

There is no DeepSeek client, no AI module, and no provider abstraction in the backend source tree. There are also no direct LLM dependencies declared in `apps/backend/package.json`.

Implication:
- Phase 2.1 must introduce a dedicated AI module/service and should do it behind a small provider interface instead of scattering HTTP calls.

#### 2. No persistent sales-agent state

`prisma/schema.prisma` currently only has:
- `User`
- `Message`

There is no table/model for:
- lead or conversation stage
- structured commercial brief
- quotation draft
- owner feedback / review status
- approval gate state

Implication:
- raw transcript alone is not sufficient; Phase 2.1 needs domain persistence.

#### 3. No Phase 2 FSM implementation is visible yet

The roadmap defines a Redis-backed FSM in Phase 2, but the current codebase does not yet expose a bot state machine module or conversation state reducer.

Implication:
- Phase 2.1 plans must assume a future handoff boundary from Phase 2 instead of coupling directly to nonexistent code.
- The safest contract is "when base qualification completes, AI sales orchestration can be triggered for a stable `conversationId`".

## Recommended Architecture

### Pattern 1: Add an AI Sales domain module, not prompt strings inside messaging code

**What:** Introduce a dedicated backend module responsible for:
- prompt building
- provider calls
- structured parsing
- brief persistence
- quote draft persistence
- review/regeneration transitions

**Why:** `MessagingService` should stay transport-only. AI sales behavior is domain logic.

**Recommended structure:**

```text
apps/backend/src/ai-sales/
  ai-sales.module.ts
  ai-sales.service.ts
  ai-sales.orchestrator.ts
  ai-provider.interface.ts
  deepseek/
    deepseek.client.ts
  dto/
  prompts/
```

### Pattern 2: Persist a structured brief and quote draft separately from raw messages

**What:** Keep `Message` as raw transcript source of truth, but add dedicated persistence for AI state.

**Suggested entities (names can vary):**
- `CommercialBrief`
  - `conversationId`
  - `projectType`
  - `businessProblem`
  - `desiredScope`
  - `budget`
  - `urgency`
  - `constraints`
  - `status`
- `QuoteDraft`
  - `conversationId`
  - `version`
  - `formatPayload`
  - `reviewStatus`
  - `ownerFeedback`
  - `approvedAt`

**Why:** Parsing the full transcript every time is brittle and makes review state hard to audit.

### Pattern 3: Make review blocking state explicit

**What:** Add a state field that prevents any customer-facing quote send unless approval exists.

**Suggested review statuses:**
- `draft_pending_owner_review`
- `owner_changes_requested`
- `draft_ready_for_recheck`
- `owner_approved`
- `delivered_to_customer` (likely Phase 3 boundary)

**Why:** Requirement `SALES-AI-05` must be enforced structurally, not only by prompt wording.

### Pattern 4: Trigger the agent from a durable handoff event

**What:** After Phase 2 marks qualification complete, enqueue an AI sales job with `conversationId`.

**Why:** Reusing BullMQ fits the current architecture and avoids doing LLM work inline in webhook/request paths.

**Recommended flow:**

1. Base qualification flow completes.
2. Backend stores/marks qualification complete for `conversationId`.
3. Queue job `ai-sales-process`.
4. Worker reads transcript + current brief state.
5. Worker calls DeepSeek.
6. Worker updates brief + quote draft + review state.
7. Worker sends appropriate internal/customer messages depending on state.

### Pattern 5: Keep prompts versioned and testable

**What:** Store prompt builders / prompt templates as source files, with schema examples and fixtures.

**Why:** This phase is prompt-sensitive and the state already warns that DeepSeek structured output reliability in Spanish agency context is not yet proven.

**Recommended prompt layers:**
- `sales_discovery_system_prompt`
- `brief_extraction_schema_prompt`
- `quote_draft_generation_prompt`
- `owner_review_regeneration_prompt`

## Data Model Guidance

### Minimum persistent state for this phase

At minimum, this phase needs a durable record for:

1. AI sales stage per conversation
2. structured brief snapshot
3. current quote draft
4. current review status
5. owner feedback / regeneration history

Trying to defer all of this to Phase 3 would make Phase 2.1 mostly unimplementable.

### Recommended conversation state split

Do not overload the existing `Message` model with many AI fields.

Use:
- `Message` for transcript/history
- AI sales tables for stateful business facts

This separation keeps reads clean and makes later CRM views simpler.

## Implementation Risks

### Risk 1: User-specific quote format is still undefined

The user explicitly stated that the quote must follow a format they already have, but that format is not yet present in the repo.

Mitigation:
- Treat the quote template as a first-class input artifact in Plan 01.
- Build the draft model so the render format can be swapped without rewriting orchestration.

### Risk 2: Phase 2 handoff contract is not implemented yet

The roadmap says Phase 2 owns the FSM, but the codebase does not yet expose the completion signal/state.

Mitigation:
- In planning, define an explicit handoff contract using `conversationId` + a qualification-complete state, rather than depending on concrete Phase 2 files that do not yet exist.

### Risk 3: AI output reliability in Spanish sales context is unverified

This is already called out in `.planning/STATE.md`.

Mitigation:
- Require structured output validation.
- Add transcript fixtures and goldens for premium consultive tone, brief extraction, and quote draft rendering.
- Keep owner review mandatory.

### Risk 4: Review loop can accidentally become customer delivery logic

If the same service handles both "prepare for owner" and "send to client" without a strong state boundary, requirement `SALES-AI-05` becomes fragile.

Mitigation:
- Keep customer delivery explicitly out of Phase 2.1.
- Let Phase 3 own the final validated approval loop and outbound release.

## Anti-Patterns to Avoid

- Embedding DeepSeek HTTP calls directly inside `WebhooksService`, `MessageProcessor`, or `MessagingService`
- Using raw transcript replay as the only source of brief/quote state
- Treating prompt text as the only guardrail for approval blocking
- Combining "owner review" and "customer send" in the same step of this phase
- Waiting for the CRM dashboard phase to introduce all review state, leaving the agent without durable workflow data
